## 思考

在回答這類問題時，我蠻喜歡先思考「為什麼」，就是 React 的推出是為了要解決什麼呢?
FB 當時 (2013 年) 為了要解決大型應用程式的複雜 UI 狀態管理問題，推出了 React 函式庫。這裡可以思考的問題有兩個：

1. 過去大型的應用程式在 UI 狀態管理上遇到了什麼問題？
2. 為什麼 React 定位自己是函式庫而非框架？

### 傳統 UI 狀態管理的困境

傳統的 UI 狀態管理會容易造成雙向資料綁定的混亂，什麼是「雙向資料綁定」呢？這個會需要另外一篇來記錄。簡單來說「雙向資料綁定」就是當 UI 改變時，會自動地影響資料模型層面。
這容易造成效能上的挑戰，應用程式的複雜度跟可維護性大幅提升，不容易除錯以及潛在地副作用等問題。

### 為何是定位在 UI 函式庫？

官方定位 React 為 UI 函式庫的主要原因在於 React 著重在以下的面向：

- 單一職責：只專注在 View 層的渲染邏輯。
- 元件為基準單位：透過組合小元件來來讓開發者自由建立。
- 宣告式 UI：使用宣告式方法，開發人員可以描述給定狀態下的 UI 外觀。

## 核心概念是？

在解釋核心概念這種可能比較開放的問題時，最難的莫過於如何去收束，比起直接說明或許更需要技巧性地去描述。

從最廣的部分來說，React 的核心有：

- 從 Class component 改成 functional component
- Fiber 架構的重構
- 單向資料流
- Scheduler，從原本的 expirationTime 改為 Lane model
- 宣告式 UI
- Hook

以下就開始本文的重點，就上述的五大要點來逐一解釋。

### Class component 改為 Functional Component

過去的 Class component 在撰寫上有幾個挑戰，首先是：

- `this` 綁定複雜，容易出錯
- react 生命週期分散
- 程式碼體積較大，壓縮效果差
- HOC 和 render props 造成層層的 wrapper hell

改為函數程式設計的優勢有 (這裡可能會被延伸提問關於函數程式設計的問題)：

- 純函式的可預測性
- 組合優於繼承
- 較容易測試
